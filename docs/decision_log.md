# Decision Log

This log records architecture judgment for the reference implementation. It is not a release diary. Each choice below is meant to keep the repository inspectable, reproducible, and bounded enough for a reviewer to understand in a few minutes.

## Core Decisions

| Decision | Choice | Why this shape | What it avoids |
|---|---|---|---|
| Language | Python | Keeps examples readable, scripts easy to run, and package behavior easy to inspect locally. | A heavy service stack that obscures the evidence model. |
| Model behavior | Deterministic mocked LLM | Keeps replay, eval, and reconstruction checks stable without live model calls or vendor credentials. | Treating stochastic model output as proof of replay correctness. |
| Storage | SQLite | Gives a local, inspectable replay store that can be regenerated by `make verify`. | Database infrastructure, hosted services, and unreproducible local state. |
| Policy engine | Real OPA | Shows an actual policy decision record with inputs and outcomes. | A toy in-repo policy language that would weaken the permission evidence claim. |
| Demo scenario | Code-review agent | Fits agent-runtime and platform-engineering evidence while staying small enough to audit. | A broad demo that looks more impressive but is harder to verify. |
| Schema coupling | Scenario-agnostic schemas | Keeps manifest, event, permission, trace, eval, and reconstruction artifacts reusable beyond the code-review fixture. | Baking one demo's vocabulary into the core evidence model. |
| Vendor posture | Vendor-neutral core | Makes the chain inspectable without requiring a cloud-specific control plane. | Recasting the repository as a Bedrock, LangSmith, OTel, MCP, A2A, or OPA replacement. |

## Scope Decisions

The repository builds one inspectable wiring pattern across release-time and runtime evidence. The core path is deliberately narrow:

```text
release manifest -> agent-step event -> OPA-backed permission packet -> trace bundle -> SQLite replay state -> deterministic eval result -> reconstruction packet
```

It does not try to become an agent framework, tracing backend, policy engine, model gateway, deployment tool, eval platform, or compliance product.

The first vendor-specific material belongs outside the core chain. Bedrock translation maps concepts to Bedrock aliases, versions, and runtime references without making Bedrock the primary implementation target.

The Bedrock translation is intentionally documentation and mapping data, not a Python workspace package. Its standalone check script is still part of `make verify`.

The root `pyproject.toml` defines the public `operational-evidence-plane` distribution. It packages the reference implementation modules, resources, and entry points together so the public release line has one installable surface. Workspace member `pyproject.toml` files remain source and development boundaries, not independently published packages.

## Verification Decisions

`make verify` is the main local contract. It regenerates state and checks joins rather than relying on hand-inspected examples.

`make test` is a small pytest smoke-test layer over public verification scripts. It is intentionally separate from `make verify` so the default verification path stays dependency-light.

The generated SQLite file is ignored because committed generated state would make the repo noisier and less trustworthy. The source of truth is the committed artifact set plus the verification scripts.

## v0.2 Decisions

| Decision | Choice | Why this shape | What it avoids |
|---|---|---|---|
| Schema extension policy for replayable permission trace fields | Add new fields as **optional** to `tool_permission_packet.v0.schema.json` rather than minting a `v1` alongside | v0.2 fields (`scoped_credential_lifetime`, `approval_capture`, `policy_bundle_version`, `release_manifest_version`, `model_alias`, `resolved_model_version`, `model_provider`) are additive record-keeping. v0.1 records that omit them still validate. The change is a forward-compatible extension, so the convergence cost of a fresh `v1` (new digests, fresh manifest binding, parallel example sets) is not justified. `update-digests` regenerates manifest and packet hashes as part of the same release. | Splitting the schema namespace prematurely and creating two parallel example chains for a strictly additive change. |
| First framework integration adapter | Model Context Protocol (MCP) under `integrations/mcp/` as documentation, mapping data, and a standalone projection script | MCP is the highest-leverage runtime-protocol surface to demonstrate one inspectable way to translate a tool-call envelope into the OEP permission packet, including the new v0.2 fields. The adapter does not call MCP servers or vendor APIs; it stays in source-inspectable territory and matches the post-core translation pattern already used by `translations/bedrock/`. Other adapters (LangGraph, OpenAI Assistants, Bedrock conversational extension) are deferred to a later release. | Treating MCP as the runtime substrate, claiming MCP-replacement scope, or expanding to multiple framework adapters in one cycle. |
| `oep replay` placement | Top-level console script `oep` registered against `oep_verify.cli:main`, with the replay implementation in `oep_permissions.replay` | The replay reader is a thin SQLite reader over existing replay state; making it a subcommand of a single `oep` dispatcher keeps the v0.2 footprint to one new entry point and leaves room for future subcommands without adding a new package. Placing the dispatcher in `oep_verify` (the existing shared support package) avoids minting a new package for one file. | A new `oep_cli` package for one dispatcher, or a separate console script that has to be repeatedly added per subcommand. |
| Defense-in-depth schema validation for adapter and replay paths | The MCP projection script and the `oep replay` reader both validate their permission-packet output against `tool_permission_packet.v0.schema.json` before exposing it | The MCP adapter byte-compares its projection with the canonical example; a schema check additionally catches corruption of the canonical example itself. The replay reader trusts SQLite state written by the demo, but a schema check catches tampering or non-conformant rows from any future write path. | Trusting upstream writers blindly and exposing non-conformant records through user-facing surfaces. |
| Deferred from v0.2 | Counterfactual replay under a substituted historical policy bundle; LangGraph / OpenAI Assistants / Bedrock framework adapters; an evaluations harness module; a UI dashboard; multi-tenant scale tests; a production deployment surface; standardization-proposal language anywhere in repo docs | The v0.2 schema additions make counterfactual replay addable later without a v1 schema bump, so deferring keeps the v0.2 surface inspectable. The Bedrock pattern under `translations/` already shows the post-core translation discipline; one framework adapter (MCP) per release keeps the integration surface auditable. Evals, UI, multi-tenant, production, and standardization claims all expand scope beyond the repository boundary statements and would weaken the «reference implementation, not framework» posture in `public_claims.md`. | Treating v0.2 as a deployable product release; promising a vendor-replacement, standard, or compliance posture; accumulating framework adapters faster than the source-inspectable mapping pattern can stay clean. |

## v0.3 Decisions

| Decision | Choice | Why this shape | What it avoids |
|---|---|---|---|
| Counterfactual replay primitive | Add policy-bundle substitution over stored decision records, with output validated by `replay/counterfactual_replay.v0.schema.json` | The v0.2 evidence chain already joins `decision_id` to policy bundle, manifest version, model metadata, approval capture, credential lifetime, event, trace, and replay state. Re-evaluating the stored OPA input under a substituted policy is the smallest useful counterfactual operation over that record. | Re-executing the LLM call, substituting model weights, or turning the replay reader into a broad off-policy evaluation engine. |
| Counterfactual mode marker | Use `OEP_REPLAY_MODE=counterfactual` plus `oep replay --counterfactual --policy-bundle ...` | An explicit mode marker keeps read-only replay and counterfactual replay visible in CLI output and logs. It mirrors the public shape of authorization replay precedents without coupling OEP to those products. | Hidden mode switches that make it unclear whether a record came from live/read-only replay or counterfactual evaluation. |
| Non-determinism handling | Add optional `nd_builtin_cache` to the permission packet schema and inject it through `input.nd_builtin_cache` during OPA evaluation | OPA builtin calls such as time or HTTP must not be re-invoked during deterministic replay. Treating captured builtin outputs as input data keeps counterfactual replay byte-identical and testable. | Letting wall-clock time, network calls, or other non-deterministic builtins leak into the replay path. |
| Composite v0.3 decision metadata | Add optional `decision_id` metadata with permission, cost, drift, cache, and identity sub-objects | The v0.3 thesis is composition under one decision id. Keeping the object optional preserves v0.1/v0.2 validation and replay while letting v0.3 examples demonstrate multi-surface attribution. | Mutating old records into a required new shape or scattering replay surfaces across unrelated top-level fields. |
| Deterministic versus evaluative replay boundary | Mark model substitution, cache-to-fresh-call substitution, and pre-session projection as `replay_class: evaluative`; keep policy, budget, reserve accounting, cache-staleness rejection, and config diffs deterministic | The reference implementation can definitively replay rules over recorded fields, but it cannot reconstruct a stochastic response from a different provider/model. Labelling the boundary keeps the replay-divergence mitigation claim narrow. | Presenting cross-provider replay as a what-would-have-happened result. |
| Demo scope | Implement three counterfactual demos over the existing code-review fixture: compound reliability, budget-per-run cross-over, and approval-per-step escalation | The demos exercise three operational stories while preserving the one-scenario discipline. They extend the fixture with synthetic multi-step state, cost, and approval attributes rather than introducing a second agent scenario. | A broader demo portfolio, live LLM calls, networked services, or vendor-specific runtime vocabulary in the core. |
| Verification gate | Add per-CR v0.3 validation targets, `validate-composite`, `validate-backward-compat`, `validate-counterfactual-replay`, `check-replay-determinism`, and `validate-counterfactual-schema`, and wire them into `make verify` and coverage | Counterfactual replay is only useful if it is reproducible and exercised end-to-end. The gate validates each surface, checks the composed replay output, and compares SQLite state, counterfactual JSON/JSONL, and DTR JSONL bytes across runs. | Treating determinism as an informal property or relying only on unit tests that do not compare generated artifacts. |

## Reader Contract

The intended review path is:

1. Read the README top section and evidence-chain diagram.
2. Run `make verify`.
3. Inspect `docs/architecture.md` for join keys and artifact roles.
4. Inspect this decision log for why the implementation is intentionally small.
5. Check `docs/public_claims.md` before reusing external wording.

If a future change adds an integration adapter, live model call, service dependency, UI, or benchmark-like claim, it should add a new decision row explaining why that complexity belongs in scope.
